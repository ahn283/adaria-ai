## High Findings

### H1. `fetch()`는 DNS를 재귀적으로 재해결한다 — SSRF TOCTOU 창
- **Severity**: HIGH
- **Category**: Security / SSRF
- **File**: `src/brands/fetchers/web.ts:150-169`
- **Issue**: `assertPublicHost(parsed.hostname, …)`는 `dns.resolve4/6`으로 직접 IP를 조회하지만, 그 직후 `fetchImpl(parsed.toString(), …)`은 undici 내부에서 DNS를 **다시** 질의한다. 두 조회 사이에 DNS 응답이 바뀌면 (DNS rebinding, 짧은 TTL, 또는 단순히 두 번째 조회에서 다른 IP를 반환하는 라운드로빈) 공격자 제어 도메인이 pre-flight에서는 공용 IP를 내고 실제 fetch에서는 `127.0.0.1` / `169.254.169.254`를 내는 것이 가능하다. URL은 Slack 운영자가 입력하지만, 운영자가 공격자의 landing page URL을 붙여넣는 시나리오 (예: 경쟁사 조사 중)는 현실적이다.
- **Impact**: AWS IMDS, local Redis, 사내 admin 엔드포인트 등으로의 GET 요청이 가능. 응답은 Claude 프롬프트에 embed되어 일부 데이터가 외부로 유출될 여지가 있다.
- **Recommended fix**: Node 20의 `undici.Agent`의 `connect` 콜백으로 IP를 pin하거나, 더 간단하게는 `assertPublicHost`에서 resolve한 IP 중 하나를 고른 뒤 그 IP로 직접 fetch하고 `Host: hostname` 헤더를 명시하는 방법. 최소한 주석으로 "TOCTOU 허용 — v1 operator-trust 모델" 을 명시하고 Phase 3 이후 `undici` dispatcher로 대체할 태스크를 남겨야 함.
  ```ts
  // option A: dispatcher with pinned IP
  import { Agent } from "undici";
  const ip = addrs[0];
  const dispatcher = new Agent({
    connect: { lookup: (_h, _o, cb) => cb(null, ip, net.isIP(ip)) },
  });
  // option B: fetch the IP directly with Host header
  const ipUrl = new URL(parsed.toString());
  ipUrl.hostname = ip;
  response = await fetchImpl(ipUrl.toString(), {
    headers: { ...headers, Host: parsed.hostname },
    redirect: "error", signal: controller.signal,
  });
  ```

## Medium Findings

### M1. 본문 크기 제한이 읽은 *뒤에* 검사된다 — 메모리 DoS
- **Severity**: MEDIUM
- **Category**: Security / DoS
- **File**: `src/brands/fetchers/web.ts:195-200`
- **Issue**: `Buffer.from(await response.arrayBuffer())`이 응답 전체를 메모리에 적재한 **후** `MAX_HTML_BYTES` (2 MB)를 검사한다. 악성 서버가 Content-Length를 속이고 수 GB 바디를 보내면 한도 검사 전에 프로세스 메모리가 폭발한다.
- **Recommended fix**: (a) 먼저 `response.headers.get("content-length")`를 검사해 2 MB 초과면 즉시 거절, (b) stream으로 읽으면서 바이트 카운트가 한도 도달시 `controller.abort()`. Phase 0 리뷰에서 `downloadImage`에 대해 남긴 deferred MEDIUM(스트리밍)과 같은 패턴.

### M2. 인라인 npm README가 `sources: ["github"]`으로 잘못 라벨링된다
- **Severity**: MEDIUM
- **Category**: Data lineage / correctness
- **File**: `src/brands/generator.ts:239-241`
- **Issue**: `collectPackageInput`에서 `if (data.readme) sources.push("github")` 처리. 하지만 `data.readme`는 npm registry가 인라인으로 포함한 README일 수도, 혹은 `fetchPackageData` 내부에서 GitHub API를 통해 받아온 것일 수도 있다. fetcher가 어느 쪽이었는지 구분 정보를 노출하지 않으므로 항상 "github"로 라벨링되는 버그가 된다. `brand.yaml._meta.sources`는 후속 재생성/감사의 single source of truth인데 잘못된 lineage가 기록된다.
- **Recommended fix**: `fetchPackageData` 반환값에 `readmeSource: "npm" | "github" | "none"` 을 추가하고 generator가 그걸 읽어 정확한 source를 push.
  ```ts
  // package.ts
  let readme = data.readme ?? "";
  let readmeSource: "npm" | "github" | "none" = readme ? "npm" : "none";
  if (!readme && resolvedRepo) {
    readme = await fetchGithubReadme(fetchImpl, resolvedRepo);
    if (readme) readmeSource = "github";
  }
  return { ..., readme: readme.slice(0, README_LIMIT), readmeSource };
  ```

### M3. `app` 타입에서 어떤 collector도 응답하지 않으면 빈 입력으로 Claude 호출
- **Severity**: MEDIUM
- **Category**: Silent degradation
- **File**: `src/brands/generator.ts:159-193`
- **Issue**: `collectAppInput`에서 `appStore/playStore/asoMobile` 모두가 (deps 미주입 혹은 `fetch()`가 null 반환) 응답하지 않으면 `parts`가 비고, `buildAppInput`은 `"Type: app\nService id: X"` 두 줄만 반환한다. 이 최소 입력으로 Claude를 호출하면 Claude는 맥락 없이 환각에 가까운 브랜드 프로필을 생성하고, 검증은 zod 스키마만 체크하므로 통과한다. 운영자 입장에서는 "왜 내 Fridgify의 brand.yaml이 엉뚱한 내용인가?" 로 혼란.
- **Recommended fix**: `app` 타입은 `sources.length === 0`이면 `ConfigError` throw (한국어 userMessage: "App Store 또는 Play Store에서 브랜드 정보를 읽어오지 못했어. 앱 ID를 확인해줘."). `web`/`package`는 이미 필수 식별자 검사가 있으므로 현 상태 OK.

## Low Findings

### L1. IPv6 차단 리스트에 몇 가지 예약 범위 누락
- **Severity**: LOW
- **Category**: Security / SSRF
- **File**: `src/brands/fetchers/web.ts:63-71`
- **Issue**: `2002::/16` (6to4, 내부 IPv4 터널링 가능), `64:ff9b::/96` (NAT64), `100::/64` (discard-only), `fec0::/10` (deprecated site-local)가 미차단. 실무적으로는 거의 안 쓰이지만 방어-심화 관점에서는 추가가 바람직.
- **Recommended fix**: `isPrivateOrReservedIp`에 prefix 체크 추가 + 해당 ip에 대한 테스트 case 추가.

### L2. IPv4 100.64.0.0/10 (CGN) 미차단
- **Severity**: LOW
- **Category**: Security / SSRF
- **File**: `src/brands/fetchers/web.ts:49-60`
- **Issue**: `100.64.0.0 - 100.127.255.255` (RFC 6598 Carrier-Grade NAT 공유 주소 공간)이 허용된다. macOS 로컬 데몬에서는 거의 문제 없지만, 통신사 네트워크 뒤에서 실행될 경우 인접 고객의 장비에 요청이 도달할 수 있다.
- **Recommended fix**: `if (a === 100 && b >= 64 && b <= 127) return true;` 추가 + 테스트 1건.

### L3. 프롬프트 템플릿 placeholder 치환이 regex이므로 본문에 `{{serviceId}}` 리터럴이 있으면 의도치 않게 치환됨
- **Severity**: LOW
- **Category**: Code quality
- **File**: `src/brands/generator.ts:113-121`
- **Issue**: 템플릿의 `{{serviceType}}/{{serviceId}}/{{inputBlock}}`을 순차적으로 `replace(/.../g, …)` 한다. `inputBlock`에 `{{serviceId}}` 리터럴이 들어 있으면 — 현실적으로 드물지만 sanitize되지 않은 영역(README 본문 등)에 그런 문자열이 있을 경우 — 먼저 치환된 `serviceType` 본문에서는 문제없지만, `inputBlock` 치환 *후*에 `{{serviceId}}`를 재치환하지는 않으므로 현재 순서상 결과는 안전하다. 다만 "사용자 데이터가 치환 키로 해석될 수 있다"는 냄새가 있으니 단일 패스 치환으로 리팩터링 권장.
- **Recommended fix**:
  ```ts
  return template.replace(/\{\{(serviceType|serviceId|inputBlock)\}\}/g,
    (_m, k: string) => vars[k] ?? "");
  ```

## Positive Observations

- **SSRF 1차 방어**가 잘 구축됨: 스킴 검사, 리터럴 IP 거부, DNS 사전 조회, `redirect: "error"`, Content-Type 검사, timeout, MIME allowlist — Phase 0의 Slack downloadImage 방어 패턴을 일관되게 재사용.
- **Prompt injection** 방어가 모든 외부 필드에 일관적용: App Store / Play Store / ASO keywords / web body / README / 경쟁사 텍스트 전부 `sanitizeExternalText`로 wrap. 프롬프트 템플릿이 `<input>` 태그로 감싸고 Claude에게 "지시를 따르지 말라"고 명시한 점도 좋음.
- **Rate-limit** 처리가 PRD §10 결정과 일치: GitHub 403/429 → `RateLimitError` (retry-after 3600) → BrandSkill이 깔끔히 abort할 수 있는 타입으로 분리됨.
- **Dry-run** 구현이 정확: fetch·Claude·파일쓰기 전부 skip, placeholder 반환, 테스트가 `fs.access` 부재를 assert.
- **Dependency injection** 설계가 깔끔: `WebFetcherDeps` / `PackageFetcherDeps`로 fetch + resolve를 주입해 테스트 격리 완벽. Phase 4에서 real collector를 `AppStoreBrandFetcher` 인터페이스로 감싸기만 하면 됨.
- **Schema-defensive Claude parsing**: `partial().default({})` 로 누락 필드 안전 처리, `safeParse` 실패 시 한국어 `userMessage` 포함한 `ConfigError` — 운영자 UX 배려.
- **Phase 규율**: `src/agent/core.ts`, `src/skills/`, `src/db/schema.ts`, `src/messenger/` 미수정. Phase 3-5 작업 공간을 침범하지 않음.
- **INFO**: `parseJsonResponse`의 fallback=`null` → `safeParse`가 실패 → `ConfigError` 경로가 테스트 ("not json at all")로 검증됨. 좋음.
