## Critical & High Findings

### H1. DM에서 flow 연속성이 깨짐 (threadKey 계산 문제)
- **Severity**: HIGH
- **Category**: Two-mode routing / UX correctness
- **File**: `src/agent/core.ts:259`, `src/messenger/slack.ts:173-174`
- **Issue**: `threadKey = "${channelId}:${threadId ?? eventTs}"`. Slack DM에서 사용자가 "reply in thread"를 명시적으로 쓰지 않으면 매 메시지가 top-level이라 `thread_ts` 가 없고 slack.ts는 `threadId = eventTs`로 fallback한다. 그러면 각 DM 메시지의 `threadKey`가 서로 달라져서 `findActiveBrandFlow`가 항상 `null`을 반환한다.
- **Impact**: PRD §4는 "Slack 대화"를 전제로 하는데, DM E2E가 동작하지 않는다. 사용자가 매 턴 쓰레드 reply를 수동으로 선택해야 해서 §8.2 verification이 실패한다. 채널에서만 작동.
- **Recommended fix**: DM은 IM 단위(채널ID) 자체를 thread로 간주하도록 분기.
  ```ts
  // in handleMessage, near line 259
  const isDm = msg.channelId.startsWith("D");
  const threadKey = isDm
    ? `${msg.channelId}:dm`
    : `${msg.channelId}:${msg.threadId ?? msg.eventTs ?? ""}`;
  ```
  또는 `MessengerAdapter.IncomingMessage`에 `conversationKey`를 올리고 Slack 어댑터가 DM에서 `channelId`만, 채널에선 `${channel}:${thread_ts}` 를 계산해서 내려주는 형태. 테스트도 DM 케이스 추가 필요.

### H2. Active flow이 다른 skill 명령을 전부 삼킴 (escape hatch 부재)
- **Severity**: HIGH
- **Category**: Two-mode routing integrity
- **File**: `src/agent/core.ts:266-285`
- **Issue**: Mode C는 `findActiveBrandFlow`가 hit하면 무조건 `continueFlow`로 라우팅한다. 사용자가 brand 플로우 중 `@adaria-ai aso fridgify` 같은 전혀 다른 명령을 보내면 ASO로 가지 않고 현재 state의 reducer에 텍스트가 먹혀 "`app`,`web`,`package` 중 하나로 답해줘" 같은 응답이 돌아온다. 반대로 사용자는 "왜 ASO가 안 돌지?" 혼란스러워진다. PRD §4.3의 유일한 탈출구는 `취소` 토큰뿐.
- **Impact**: 한 스레드에서 brand 플로우가 열려 있으면 30분 idle까지 다른 모든 skill이 차단된다. 채널에서 여러 사람이 쓰는 경우는 문제 없지만(user-scoped), 동일 사용자가 multitask하는 경우 UX가 사실상 hang.
- **Recommended fix**: Mode C 진입 전에 "명시적 명령이면 Mode A 우선"을 허용. `SkillRegistry.findSkill(msg.text)`가 non-null이면 그쪽에 양보하고, flow는 다음 비-명령 메시지를 기다리게 한다(또는 `취소`와 동등하게 플로우를 종료).
  ```ts
  const explicitSkill = this.skillRegistry.findSkill(msg.text);
  if (activeFlow !== null && explicitSkill === null) {
    // current Mode C path
  } else if (explicitSkill) {
    // Mode A — optionally delete the abandoned flow row here
  }
  ```
  최소한 ASK_TYPE/ASK_IDENTIFIER에서 새 명령 토큰(`aso`, `review`, `brand` 제외 등)이 들어오면 `취소됨 + Mode A 재-dispatch` 로 escape 시키는 것. 테스트도 추가.

## Medium & Low Findings

### M1. `COLLECTING → PREVIEW`에서 `brand.yaml`이 조기에 디스크에 기록됨
- **Severity**: MEDIUM
- **Category**: Write-path safety / flow semantics
- **File**: `src/skills/brand.ts:145-172`, `src/brands/generator.ts:369-376`
- **Issue**: `runCollection`이 `generateBrandProfile`을 호출하는데 generator는 `brand.yaml`을 바로 `fs.writeFile`한다. PREVIEW 단계에서 사용자가 `취소`하면 row는 지워지지만 yaml 파일은 디스크에 남는다. PRD §4.3은 PREVIEW 이후 `저장` 버튼을 눌러야 "저장"된다는 모델처럼 읽힌다 (그래서 상태명도 ASK_LOGO로 넘어간 뒤 DONE이 "저장 완료"라고 말함).
- **Impact**: v1에서는 파일이 남아도 다음 실행에서 덮어쓰이므로 데이터 무결성 문제는 없지만, 사용자가 "취소했는데 파일이 있네" 혼란을 느낄 수 있고 PRD 문구와도 불일치. `ADARIA_DRY_RUN=1`에서는 generator가 쓰지 않으므로 OK.
- **Recommended fix**: 세 가지 옵션 중 선택. (a) generator API를 `{ profile, write: () => Promise<string> }` 형태로 바꿔 BrandSkill이 PREVIEW 승인 후에만 호출. (b) CANCELLED 전이에서 `brandsDir(serviceId)/brand.yaml` 삭제(logo/design도). (c) PRD 문구를 "PREVIEW 단계에서 이미 저장됨, `저장`은 이미지 추가 단계로 진입" 으로 명시. 가장 원칙적인 건 (a).

### M2. `COLLECTING` 응답이 사용자에게 도달하지 않음
- **Severity**: MEDIUM
- **Category**: UX
- **File**: `src/skills/brand.ts:126-139`
- **Issue**: reducer가 "좋아. 웹사이트 분석 중이야… 잠시만." 를 반환하지만 BrandSkill이 즉시 `runCollection`을 체이닝하며 그 reply를 버린다. 외부 fetch + Claude 호출이 10–30초 걸릴 수 있어 사용자 입장에서 턴이 멈춘 듯 보인다. status message는 "🤔 Thinking..."로만 보여짐.
- **Impact**: 체감 지연. 특히 package 타입은 npm+GitHub+Claude까지 붙어 긴 편.
- **Recommended fix**: (a) reducer 전이 응답을 별도 메시지로 먼저 post하고, generator 완료 후 상태 메시지를 PREVIEW 내용으로 update. (b) 또는 core.ts에 "intermediate status"를 흘려보낼 수 있는 콜백을 `SkillContext`에 추가. (a)가 이번 단계 변경 폭이 작다.

### M3. `runCollection`의 ternary 식이 no-op (죽은 코드)
- **Severity**: MEDIUM
- **Category**: Code quality
- **File**: `src/skills/brand.ts:169`
- **Current code**:
  ```ts
  data: { ...data, serviceId: result.profile._meta.serviceType === serviceType ? serviceId : serviceId },
  ```
- **Issue**: 양 branch가 같은 `serviceId`. 원래 의도가 있었는지 모호. 리뷰어에게도 혼동을 준다.
- **Recommended fix**: `data: { ...data, serviceId }` 로 단순화. 의도가 "generator가 yamlPath 경로로부터 id를 바꿨을 수 있으니 덮어쓰기"였다면 `result.yamlPath`에서 파싱하도록 명시적으로 수정.

### M4. 동일 사용자의 멀티-쓰레드 동시 flow 테스트 부재
- **Severity**: MEDIUM
- **Category**: Test coverage
- **File**: `tests/skills/brand.test.ts`
- **Issue**: PRD §4.2는 unique index가 `(user_id, thread_key)`라 "동일 사용자가 다른 쓰레드에서 독립 flow"를 명시적으로 지원. 현재 9개 테스트 모두 단일 thread. 두 개의 threadKey에서 병행 dispatch → 각각 독립 row, 각각 CANCELLED가 상대를 건드리지 않음을 검증하는 테스트가 없다.
- **Recommended fix**: `threadKey: "C1:aaa"` / `"C1:bbb"` 로 각각 dispatch → 두 row 확인 → 한쪽만 CANCELLED → 다른 쪽은 여전히 active. 3–5줄짜리 케이스 추가.

### L1. Idle cutoff 재사용이 의미 과적
- **Severity**: LOW
- **Category**: Config-driven design
- **File**: `src/agent/core.ts:659`, `src/db/queries.ts:884` (별도 `deleteStaleBrandFlows` 존재)
- **Issue**: `safety.approvalTimeoutMinutes`(Approval용 30분 기본값)를 flow idle cutoff로 재사용. Approval timeout을 10분으로 줄이면 brand flow도 덩달아 10분으로 줄어든다. `brand_flows.idleMinutes` 또는 `brand.idleTimeoutMinutes` 같은 전용 설정을 두는 편이 config-driven 원칙에 맞다.
- **Recommended fix**: `AdariaConfig.brand?.idleMinutes ?? 30` (후방호환 default 30). `queries.ts`의 stale cleanup도 같은 값을 쓰도록.

### L2. `record.metadata`는 read_only여도 재저장됨
- **Severity**: LOW
- **Category**: Data handling
- **File**: `src/skills/brand.ts:113-136`
- **Issue**: `continueFlow`가 `record.metadata`를 upsert 파라미터로 다시 쓴다(`buildResult` → `writeFlowRow`). user_id/thread_key는 불변이므로 재쓰기는 낭비고, 로그에서 "metadata가 어디서 왔는지" 추적을 미묘하게 흐린다.
- **Recommended fix**: `writeFlowRow`에서 기존 row의 user_id/thread_key는 `ON CONFLICT DO UPDATE`가 변경하지 않으므로(현재도 그렇다) 기능상은 OK. 다만 함수 주석에 "user_id/thread_key는 ON CONFLICT 업데이트 대상 아님" 명시.

### L3. `flowId` 엔트로피 및 충돌 확률
- **Severity**: LOW
- **Category**: Naming / data
- **File**: `src/skills/brand.ts:369-371`
- **Issue**: `crypto.randomBytes(9)` = 72 bit. 통계적으로 충분하지만 PRD §4.2는 "ULID"를 명시. 혼재. base64url 인코딩이라 시간 정렬이 불가능해 디버깅 시 "이 flow가 언제 시작됐는지" audit log cross-ref이 조금 불편.
- **Recommended fix**: `ulid` 라이브러리 도입이 과하면 `${Date.now().toString(36)}_${randomBytes(6).toString("base64url")}` 같은 time-sortable 포맷. 또는 PRD 문구를 "opaque id"로 완화.

## Data Flow Issues

- **Slack event → Mode C**: `IncomingMessage.images` → `ContinuationMessage.files`로 그대로 전달 (core.ts:281). 타입 호환 OK. 다만 core의 `buildSkillContext`가 `downloadFile` 클로저를 만들 때 `authHeader`/`filename`를 조건부로 spread하는 수고를 하지만, 어차피 원본 객체는 동일 필드를 optional로 갖는다. M6.7 Phase 0에서 이미 `ImageAttachment`와 `downloadFile` 시그니처가 호환되므로 spread 없이 passthrough 가능 (`ctx.downloadFile = this.messenger.downloadImage.bind(this.messenger)`). 현재 방식도 `exactOptionalPropertyTypes`에 맞춰 의도적으로 작성된 것으로 보여 변경 필수는 아니나 단순화 여지 있음 (INFO).
- **Mode C → Mode A 우회 불가**: H2와 동일 이슈, 데이터 경로 측면에서 보면 "Mode C는 Mode A/B를 완전히 block한다". 의도된 설계지만 명령 escape hatch 누락.
- **Approval plumbing 통합**: `skillResult` 캡처를 Mode A/Mode C 공통으로 빼낸 리팩터링은 깨끗하다. BrandSkill은 approvals이 비어있지만, 향후 multi-turn skill이 승인 gate를 갖는 경우도 같은 경로를 탄다. 기존 7 skills의 approval UX 회귀 없음을 확인 (`aso`, `seo-blog`, `social-publish`의 write path는 모두 `skillResult.approvals`를 통해 동일 loop 진입).

## Two-mode routing integrity

- **BrandSkill은 MCP tool로 등록되지 않았다**: `mcp-manager.ts`의 `registerTool` 호출은 db-query / collector-fetch / skill-result / app-info 넷뿐. `skills/registry.ts`에만 추가. 불변식 유지 ✓.
- **Mode A의 Mode B fall-through는 정상**: `activeFlow === null` 경로에서 `findSkill` → null → `invokeClaudeWithContext` 진입. 기존 경로 회귀 없음 ✓.
- **`flowContext` 주입이 기존 7 skills에 미치는 영향**: 각 skill의 `dispatch(ctx, text)`는 `ctx.flowContext`를 읽지 않는다 (`aso`, `seo-blog`, `review`, `onboarding`, `short-form`, `sdk-request`, `content`, `social-publish` 전부 미참조 확인). double-dispatch 위험 없음 ✓.
- **`onApprovalResolved`의 `buildSkillContext()` no-arg 호출**: `flowContext` undefined로 빌드, `downloadFile`은 여전히 포함됨. Approval executePost 경로에서는 flow 정보가 필요 없으므로 OK. 회귀 없음 ✓.

## Positive Observations

- **Reducer / persistence / dispatch 분리**: `flow.ts`는 완전히 순수(no DB, no clock), `BrandSkill`이 persistence와 side-effect를 소유, `core.ts`는 라우팅만. Phase 3 리뷰에서 칭찬한 설계가 Phase 4에서도 일관.
- **approval 루프 통합 리팩터링**: Mode A/C를 `skillResult` 변수로 통일한 건 다음 multi-turn skill이 나올 때 자연스럽게 확장 가능.
- **MIME whitelist + path-traversal 가드**: `saveImage`가 `ALLOWED_IMAGE_EXTS` 검증 + `removeExistingImages` 관리 + `brandsDir`의 `SERVICE_ID_PATTERN` 보호로 삼중 방어. 생성자 측 `downloadImage`도 MIME/size/host 재검증하므로 defense-in-depth.
- **`ADARIA_DRY_RUN`은 generator 레이어에서 이미 처리**: BrandSkill 쪽에서 별도 가드 없이도 안전. PRD §4.6 준수.
- **Test file의 `downloadFile` 모킹**: Slack 의존성을 끊고 BrandSkill의 파일 저장 분기를 빠르게 검증. `authHeader`까지 타는 end-to-end는 `tests/messenger/slack.test.ts`가 별도로 커버.
- **테스트가 write-path까지 검증**: "bad MIME reject" 케이스가 `downloadFile`이 호출되지 않음을 명시적으로 확인. 회귀 방지에 중요.
