## Medium Findings

### M1. `brandsDir()` serviceId Unicode/control-char allowlist 누락
- **File**: `src/utils/paths.ts:42-64`
- **Issue**: 차단하는 건 `/`, `\`, `\0`, `.`/`..`, 앞이 `.`인 것 뿐. `:` (Windows/APFS resource fork), 개행(`\n`/`\r`), NBSP 같은 zero-width, URL-encoded `%2e%2e` 등은 통과. Phase 4에서 serviceId가 사용자 자유입력(서비스 식별자)으로 들어오면 파일시스템 부작용 가능.
- **Impact**: 현재는 traversal은 막히지만 파일시스템에 이상한 파일명 생성, 로그 스푸핑, macOS extended attribute 혼선 여지.
- **Fix**: 화이트리스트 정규식으로 전환.
  ```typescript
  if (!/^[a-z0-9][a-z0-9._-]{0,63}$/i.test(serviceId)) {
    throw new Error(`invalid serviceId: ${serviceId}`);
  }
  ```

### M2. `findImageFile` 심링크/디렉터리 구분 부재
- **File**: `src/brands/loader.ts:129-153`
- **Issue**: `fs.readdir` 결과가 파일인지 심링크인지 체크 없이 `fs.readFile`로 감. 악의적 `logo.png → /etc/passwd` 심링크가 `~/.adaria/brands/{serviceId}/`에 있으면 base64로 Claude 프롬프트에 삽입 가능. 단일 사용자 로컬 에이전트라 공격 표면은 작지만, PRD §7 “fixed output path / no traversal” 정신과 일관되게 막는 편이 좋음.
- **Fix**: `{ withFileTypes: true }` + `entry.isFile()` 체크, 또는 `fs.lstat` 후 `isSymbolicLink()` 거부.

### M3. 테스트 — 잘못된 YAML 문법 / 큰 파일 / 심링크 엣지 미커버
- **File**: `tests/brands/loader.test.ts`
- **Issue**: 1) `yaml.load` 파싱 실패(`: : :` 같은 깨진 YAML) 경로 테스트 없음 — 현재 코드에선 `YAMLException`이 그대로 날아가며 `ConfigError`로 래핑 안 됨. 2) `brandsDir` 생성자 테스트에 NUL(`\0`), 개행 문자 케이스 없음. 3) 알 수 없는 `_meta` 필드가 섞인 YAML의 strict/strip 동작 미확인(Zod 기본 strip이라 silently drop — 의도적인지 명시 필요).
- **Fix**: `yaml.load` throw 케이스를 잡아 `ConfigError`로 래핑하고 테스트 추가. `brandsDir("a\0b")`, `brandsDir("a\nb")` 케이스 추가.

## Low Findings

### L1. `brandsDir()`의 `ADARIA_HOME` 재조회 분기 중복
- **File**: `src/utils/paths.ts:43-46`
- **Issue**: `BRANDS_DIR` 상수는 모듈 로드 시 고정, 함수는 `process.env["ADARIA_HOME"]`를 재조회. 두 경로가 달라질 수 있음(테스트 환경에선 환경변수를 나중에 바꿈). 의도는 맞지만 `BRANDS_DIR` export와 의미가 엇갈려 혼동 유발. 주석으로 명시하거나 `BRANDS_DIR`은 initial default로만 남기고 helper 우선 사용 문서화.

### L2. `formatBrandContext`에서 `visual.primaryColor`만 있고 style 없는 경우 헤더 출력
- **File**: `src/brands/loader.ts:119-124`
- **Issue**: `style`이 비고 `primaryColor`만 있을 때 "Visual:" 섹션 헤더가 나오고 style은 생략. 현재 생성 프롬프트는 primaryColor를 수동 편집 필드로 쓰는 PRD §3 정책과 부합하므로 동작 자체는 맞음. 다만 헤더 아래 1줄짜리 섹션이 일관돼 보이도록 순서(color → style)를 PRD 표기와 맞춰 재정렬 고려.

### L3. `BrandImage.data`는 base64 문자열이지만 필드명이 관행과 어긋남
- **File**: `src/types/brand.ts:91-93`
- **Issue**: Anthropic vision content block은 `{ type: "base64", media_type, data }` 구조를 그대로 씀. 현재 이름 `data`는 맞지만 JSDoc이 “Base64-encoded file bytes”라고만 적혀 있어 나중에 Buffer로 바꾸는 리팩터 시 충돌 가능. 타입을 `data: string & { readonly __brand: "base64" }` 브랜디드 타입으로 명시하거나 interface 최상단에 “Never pass raw Buffer here” 주석 추가.

## Positive Observations

- `loadBrandProfile`의 ENOENT → null 경로가 PRD §1.3 graceful degradation을 정확히 구현. `formatBrandContext(null) === ""`와 합쳐서 `{{brandContext}}` 치환이 항상 안전.
- Zod `default()`를 섹션/필드 양쪽에 둬서 사용자가 _meta만 있는 최소 YAML을 써도 포맷터까지 null-safe로 흐름.
- `brandsDir()`가 `ADARIA_HOME`을 call-time 재조회 — 테스트에서 env 오버라이드만으로 격리 가능 (mkdtemp 패턴이 깔끔).
- `ConfigError`로 래핑하며 `userMessage`에 재생성 명령(`@adaria-ai brand`)까지 친절히 유도 — 운영자 UX 우수.
- `yaml.load`에 `JSON_SCHEMA` 지정 — YAML 고급 태그(`!!js/function` 등) 로드 경로 차단. `safeLoad` 동급 방어.
- 19개 테스트 커버리지 탄탄, 확장자 variant까지 `it.each`로 나열 — Phase 2 확장 시 regression 포인트로 유용.
