## High Findings

### H1. `downloadImage`가 URL scheme/host을 전혀 검증하지 않음 — 제한적 SSRF 벡터

- **Severity**: HIGH
- **Category**: Security / SSRF
- **File**: `src/messenger/slack.ts:271`
- **Issue**: `fetch(attachment.url, { headers })`에 들어가는 `attachment.url`이 임의 scheme(`file://`, `http://169.254.169.254/...`, `http://localhost:...`)일 수 있다. 현재 코드는 scheme/host 검증이 없고, `Authorization: Bearer <botToken>` 헤더가 그대로 붙는다.
- **Impact**:
  1. **토큰 유출**: 공격자가 `event.files[].url_private`에 외부 HTTP URL을 주입할 수 있다면(이론상 Slack 서버가 통제하지만, 만약 모킹/테스트/타 어댑터 재사용시), Bearer 토큰이 제3자 서버로 누출된다.
  2. **내부 네트워크 프로빙**: `downloadImage`가 어떤 경로로든 사용자 제어 URL(brand-profile Phase 2의 웹 페처 재사용 등)을 받게 되면 `http://169.254.169.254/latest/meta-data/` 같은 AWS IMDS를 찌를 수 있다. PRD.md §3에서 BrandSkill의 web fetcher는 "safe-fetch allowlist / private IP block"을 요구한다 — 같은 원칙이 `downloadImage`에도 필요하다.
  3. **`file://` 지원 여부**: Node 20 `fetch`(undici)는 기본적으로 `file://`를 차단하지만, Node 업그레이드나 polyfill 시 리그레션 가능성이 있다.
- **Current code**:
  ```typescript
  const res = await fetch(attachment.url, { headers });
  ```
- **Recommended fix**: URL 파싱 + scheme/host 허용 목록 가드. Slack 호스트만 허용하면 Phase 4의 BrandSkill이 이를 "slack 파일 전용"으로 안심하고 쓸 수 있고, 나중에 다른 출처가 필요하면 별도 메서드로 분리된다.
  ```typescript
  // Allow only https://files.slack.com — all Slack `url_private` URLs
  // resolve to this host. This boxes downloadImage in to its one job
  // and prevents attacker-controlled URLs from being fetched with our
  // bot token in the Authorization header.
  const SLACK_FILES_HOST = "files.slack.com";
  let parsed: URL;
  try {
    parsed = new URL(attachment.url);
  } catch {
    throw new Error(`downloadImage: invalid URL ${attachment.url}`);
  }
  if (parsed.protocol !== "https:") {
    throw new Error(`downloadImage: non-https URL rejected: ${parsed.protocol}`);
  }
  if (parsed.hostname !== SLACK_FILES_HOST) {
    throw new Error(
      `downloadImage: host ${parsed.hostname} not allowed (expected ${SLACK_FILES_HOST})`,
    );
  }

  const res = await fetch(parsed.toString(), { headers, redirect: "error" });
  ```
  `redirect: "error"`도 함께 — Slack이 HTTP→HTTPS나 CDN으로 리다이렉트한다면 새로운 호스트에 Bearer 토큰이 따라가는 걸 끊어준다(undici 기본은 follow). 그리고 테스트에 `rejects non-slack host`, `rejects non-https scheme`, `rejects redirect` 3개 추가.

## Medium Findings

### M1. Content-Length 선언값이 실제 본문과 달라도 버퍼 경계 전까지 신뢰된다 — streaming이 아니라서 큰 문제는 아니지만 메모리 spike 가능

- **Severity**: MEDIUM
- **Category**: Correctness / Resource exhaustion
- **File**: `src/messenger/slack.ts:292-307`
- **Issue**: 현재 로직은 (1) Content-Length가 있으면 그 값으로 한 번 거르고, (2) `arrayBuffer()`로 전체 본문을 메모리에 올린 뒤 재검사한다. 악의적/버그 있는 서버가 Content-Length를 `1024`로 선언하고 실제로 500MB를 보내면, `arrayBuffer()`가 500MB를 메모리에 쌓을 때까지 가드가 작동하지 않는다. daemon이 long-running이고 macOS 데스크톱 프로세스라 OOM 위험은 낮지만, 다중 동시 업로드가 발생하면 실질적 DoS 벡터가 된다.
- **Impact**: 단일 사용자 daemon이므로 실제 공격 표면은 매우 작다. 그러나 (a) M6.7 Phase 4에서 사용자가 여러 장의 이미지를 한 번에 보내거나, (b) Slack 파일 서버가 오동작하여 큰 본문을 보내는 상황에서 daemon 메모리가 튀어 KeepAlive 리스타트 → launchd 루프 가능성.
- **Current code**:
  ```typescript
  const buffer = Buffer.from(await res.arrayBuffer());
  if (buffer.byteLength > MAX_IMAGE_BYTES) { ... }
  ```
- **Recommended fix**: 스트리밍 리더로 바꾸고 누적 바이트가 cap을 넘으면 즉시 중단. 5MB cap이 작아서 스트리밍 복잡도 vs. 이득 tradeoff는 borderline이지만, 방어 코드로서는 가치 있다.
  ```typescript
  // Stream and enforce the cap incrementally so a misbehaving server
  // can't force us to buffer half a gigabyte before we bail.
  if (!res.body) {
    throw new Error("downloadImage: response has no body");
  }
  const chunks: Uint8Array[] = [];
  let total = 0;
  const reader = res.body.getReader();
  for (;;) {
    const { done, value } = await reader.read();
    if (done) break;
    total += value.byteLength;
    if (total > MAX_IMAGE_BYTES) {
      await reader.cancel();
      throw new Error(
        `downloadImage: stream exceeded ${String(MAX_IMAGE_BYTES)} byte cap`,
      );
    }
    chunks.push(value);
  }
  const buffer = Buffer.concat(chunks, total);
  ```
  또는 단순히 "현 구현으로 충분" + 테스트에 `declared 작음 + 실제 큼` 케이스 추가하고 TODO 남기는 것도 현재 스코프에서는 수용 가능. 최소한 후자는 해줄 것을 권장.

### M2. `writeFile` 쓰기 원자성 없음 — 중간 실패시 부분 파일이 남아 다음 스킬 호출이 잘못된 이미지를 읽을 수 있다

- **Severity**: MEDIUM
- **Category**: Correctness / Atomicity
- **File**: `src/messenger/slack.ts:309-310`
- **Issue**: `writeFile(normalized, buffer)`는 atomic하지 않다. `buffer`가 메모리에 완전히 있으므로 현재 구현에서는 "부분 파일"이 생길 확률이 아주 낮지만(fsync까지 실패하는 경우에만), 디스크 full 같은 상황에서 0바이트나 잘린 파일이 남을 수 있다. BrandSkill(Phase 4)이 `brandsDir(serviceId)/logo.png`를 읽을 때 "파일이 존재한다 ≠ 유효한 이미지"라는 가정은 밖에서 책임지게 될 텐데, 지금 설계에서는 이 primitive가 atomic write를 해주면 호출자 계약이 깔끔해진다.
- **Impact**: 낮음. 단일 유저, 디스크 full 빈도 낮음. 하지만 `logo.png`가 0바이트로 저장되면 이후 vision 호출이 조용히 실패하고 디버깅이 어렵다.
- **Recommended fix**: `${normalized}.tmp`에 쓰고 `rename`으로 atomic 교체.
  ```typescript
  const tmp = `${normalized}.tmp`;
  await mkdir(dirname(normalized), { recursive: true });
  await writeFile(tmp, buffer);
  await rename(tmp, normalized);
  ```
  `node:fs/promises` `rename`은 같은 파일시스템 안에서 atomic. `brandsDir`는 `~/.adaria/` 하위라 같은 FS가 보장된다.

## Low Findings

### L1. 경로 탐색 가드가 빈 세그먼트(`..`만 매칭)는 잡지만 인코딩된 형태(`%2e%2e`)나 Windows 드라이브 루트는 다루지 않는다

- **Severity**: LOW
- **Category**: Security / Path traversal (edge cases)
- **File**: `src/messenger/slack.ts:258`
- **Issue**: `destPath.split(/[\\/]/).includes("..")`는 다음을 잡는다:
  - `/tmp/foo/../bar` ✓
  - `/tmp/..` ✓
  - Windows 세퍼레이터 `\..\` ✓

  다음은 잡지 못한다:
  - `/tmp/...` (세 점) — filesystem 상 유효한 디렉토리명이므로 OK, 실제 피해 없음
  - `/tmp/%2e%2e/evil` — URL 인코딩은 fs에서는 리터럴 문자열이라 상위로 올라가지 않음. OK.
  - 심볼릭 링크를 통한 우회 — `destPath` 자체는 안전하나 `dirname(normalized)`이 심링크로 다른 곳을 가리키는 경우. Phase 0에서는 callsite 부재라 이론적이고, Phase 1에서 `brandsDir()` 도입 시 `~/.adaria/brands/`가 유저 소유 디렉토리라 실질 위험 낮음.
- **Impact**: 현재 컨택스트에서 실제 exploit 경로 없음. 하지만 향후 `destPath`가 더 신뢰도 낮은 데서 오면 재검증 필요.
- **Recommended fix**: Phase 1에서 `brandsDir()`가 생기면 `downloadImage` 위에서 `resolve(destPath).startsWith(resolve(brandsDir()))` 같은 prefix check를 callsite에 추가. 이 primitive 자체는 지금처럼 놔둬도 충분 — 방어 계층은 callsite가 책임지는 쪽이 맞다(Adapter는 메신저-agnostic한 가드만).

## Info / Positive Observations

### I1. 입력 기준 `..` 검사 — 정확한 판단

`normalize()`가 `..`를 lexically 줄이는 특성을 코멘트(L254-257)에서 정확히 지적하고 normalize *이전* 원본을 검사하는 구조가 옳다. 많은 프로젝트가 이 지점에서 실수한다. 👍

### I2. Content-Type cross-check — files:read 누락 케이스까지 고려됨

Slack이 scope 부족시 HTML 에러 페이지를 200으로 돌려주는 패턴을 `contentTypeFirstToken` 화이트리스트로 잡은 것(L281-290) — 실제 운영 중 자주 만나는 함정을 선제적으로 커버했다. 테스트(`fails closed when the server returns text/html`)까지 있어서 회귀 방어선도 갖춰졌다.

## Data Flow Issues

### `app_mention` 파일 포워딩 — 중복 처리 위험 검토

변경 후 `app_mention`과 `message` 핸들러 둘 다 `event.files`를 forward한다. 중복 처리 위험을 전체 흐름 기준으로 추적:

```
DM 파일 업로드      → Slack이 `message` 이벤트만 발송 (app_mention은 DM에서 발송 안 됨)
Channel @mention + 파일 → Slack이 `message`(channel_type="channel") + `app_mention` 둘 다 발송
  - message 핸들러: msg.channel_type !== "im" 가드(L139)로 early return
  - app_mention 핸들러: 정상 처리
  → 실제로는 app_mention 경로로만 진입
Channel 파일 업로드(mention 없음) → `message`만 발송, `channel_type="channel"` → 핸들러 early return → 무시
```

**결론**: 중복 처리 경로 없음. `isDuplicate(ts)`가 2차 방어선으로 있고, `message` 핸들러의 `channel_type !== "im"` 가드가 1차 방어선. 정상.

한 가지 주의: `app_mention`에서 `isDuplicate` 체크가 **파일 처리 앞**에 있어서(L173) 만약 Slack이 동일한 `event.ts`로 event만 재전송한다면 두 번째는 무시된다 — OK.

## Positive Observations

1. **방어 순서가 옳다** — destPath 검증(absolute, `..`) → MIME 검증 → fetch → content-type 재검증 → content-length → 실제 크기. 네트워크 호출 전에 인자 검증이 먼저 일어나므로 불필요한 Slack 호출이 발생하지 않는다.
2. **테스트 커버리지가 구체적** — happy path 1개 + 실패 경로 7개로 각 가드가 회귀 방어선을 갖는다. 특히 `content-length` 선언값과 실제 본문이 다른 케이스(sneaky.png)를 분리해서 커버한 점이 좋다.
3. **`SlackFile` 인터페이스 호이스팅** — 이전에 `app.message` / `extractImages` / 곧 추가될 `app_mention` 세 군데에 같은 anonymous shape가 있던 걸 하나로 묶었다. DRY 관점에서 깔끔.
4. **`ImageAttachment.authHeader`의 Slack-specificness에 대한 주의** — "리뷰 포커스 4"에서 질문한 점: 현재 추상화 수준은 적절하다. "인증 헤더를 주면 fetch에 붙인다"는 일반화(Bearer/Basic/Custom 모두 수용)이고, Telegram 등 다른 메신저도 인증 토큰이 필요할 수 있다(Telegram Bot API의 `getFile` → `file_path` → `https://api.telegram.org/file/bot<token>/<file_path>` 패턴). `authHeader`는 충분히 generic. ✓
5. **`downloadImage?` optional signature** — 비-Slack 어댑터가 구현 부담 없이 진입 가능. Telegram 어댑터가 들어올 때도 깔끔하게 선택 구현 가능. ✓
